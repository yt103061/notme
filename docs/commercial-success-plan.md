# not me 商業化に向けた UI 刷新・ランダム対戦実装プラン

## 目的

GitHub に push すると Vercel に自動デプロイされる現在の運用を前提に、`not me` を「遊べるプロトタイプ」から「商業的に継続運用できる対戦ゲーム」へ引き上げるための実装計画を整理する。

対象は大きく次の2つ。

1. UI・インタラクションの大幅刷新
2. ランダムオンライン対戦の実装

## 現状把握

### プロダクトの強み

- ルールの核は明確。自分だけ見えない `not me` カード、相手の反応からの推理、同時ベット、交換・奪取による事件性がある。
- ゲームエンジンは `src/engine/` に分離され、Supabase Edge Function 側でも再利用されているため、UI とサーバー権威ロジックを分けて改善しやすい。
- ルームコード式オンライン対戦は実装済みで、`online_rooms.mode` にはすでに `room` / `queue` の概念がある。ランダムマッチング用の土台は存在する。
- ウォレット、ランキング、匿名アカウント、PWA、共有導線など、商業運用に必要な周辺機能の下地がある。

### 現状 UI の問題

添付スクリーンショットと公開 URL の現状から、主な問題は「情報量が多いこと」そのものではなく、情報の優先順位・状態変化・次に押すべき行動が視覚的に整理されていないことにある。

- 画面上に、相手席、相手 not me、チップ、場札、ポット、自分の手札、自分の not me、ヒント、ベットボタンが同時に露出しており、視線の着地点が多い。
- 「今は何フェーズか」「自分は何を決めるべきか」「相手は何をしたか」がモーダルやラベル中心で処理され、盤面上の出来事として伝わりにくい。
- チップが単なる数値・ラベルに見え、賭けた／奪った／失ったという快感や痛みが弱い。
- 交換・奪取がゲームの山場なのに、演出が軽いため、プレイヤーが状況を理解する前に状態だけが変わって見える。
- ゲーム終了後のホーム復帰、もう一度、オンラインへ移動、共有などの導線が商業ゲームとして不足している。

### オンライン対戦の現状

- ルームコードの作成・参加・開始は `OnlineLobby` で実装されている。
- クライアント API は `createOnlineRoom` / `joinOnlineRoom` / `startOnlineRoom` / `fetchOnlineView` など、Edge Function への薄いラッパーになっている。
- Edge Function `online-room` には `create` の `mode: 'queue'` 受け入れがすでにあるが、待機中の queue ルームを探して自動参加する `matchmake` アクションは未実装。
- DB スキーマ上も `online_rooms.mode` は `room` / `queue` を許容しており、ランダム対戦は既存テーブルを拡張せずに開始できる可能性が高い。

## 方針

### 1. UI は「情報を減らす」のではなく「フェーズごとに主役を切り替える」

本作はカード、相手、場札、チップ、ヒントのすべてが推理材料なので、単純に情報を隠しすぎるとゲーム性が落ちる。刷新方針は、常時すべてを同じ強さで見せるのではなく、フェーズごとに主役を切り替えること。

- 配札直後: 自分の見えている手札と、相手のリアクションを主役にする。
- ベット判断: 自分のヒント、場札、ポット、相手の直前行動を主役にする。
- 交換・奪取: 対象選択とリスク説明を主役にする。
- ショーダウン: 自分の `not me` 開示と役判定を主役にする。
- 終了後: 勝敗、チップ増減、次の行動を主役にする。

### 2. ランダム対戦は MVP を小さく作り、待ち時間体験を厚くする

ランダム対戦は「存在すること」が価値になる一方、初期ユーザー数が少ない段階では待ち時間が最大の離脱要因になる。最初から高度なレーティングや地域分割を入れず、以下を優先する。

- まずは 2〜4人の公開キューに自動参加できること。
- 一定秒数で人数不足でも開始できること。
- 待機中はキャンセル可能にすること。
- マッチしない場合は AI 混入、ルームコード共有、CPU 戦への退避を提示すること。

  - `home_start_random_match`
  - `matchmaking_start`

  - `matchmaking_cancel`

  - `showdown_complete`
  - `result_home_click`

- KPI ダッシュボード用に最低限、ブラウザコンソール・Supabase ログ・将来の分析 SDK のどれにも差し替えられる抽象層にする。

### 完了条件

- プレイヤーがどこで離脱しているか、ホーム・マッチング・ゲーム中・リザルトの単位で追える。

## Phase 1: ホームとリザルトの導線刷新

### 目的

「始める」「オンラインで遊ぶ」「戻る」「もう一度」が迷わず選べる商業ゲームの基本導線を作る。

### 実装

- ホーム画面を次の3 CTA に再構成する。
  - すぐ遊ぶ: AI 戦
  - ランダム対戦: 公開キュー
  - 友だちと対戦: ルームコード
- オンライン対戦ボタンの先で、ルームコードだけでなくランダム対戦を並列に提示する。
- リザルト画面に以下を必ず配置する。
  - ホームへ戻る
  - もう一度遊ぶ
  - ランダム対戦へ
  - 共有
- オンライン対戦終了時にもホームに戻れるようにする。

### 完了条件

- どの終了状態からでも 1 タップでホームへ戻れる。
- AI 戦、ルームコード、ランダム対戦の入口が明確に分かれる。

## Phase 2: 盤面 UI の情報設計刷新

### 目的

ごちゃつきの主因である「全要素の同時主張」を解消し、各フェーズで見るべきものを明確にする。

### 実装

#### 盤面レイヤー

- 背景、相手席、場札、自分の手札、操作パネル、通知をレイヤーとして分離する。
- 操作パネルは常時固定ではなく、フェーズに応じて内容を差し替える「コマンドドック」にする。
- 相手席は詳細を常時表示せず、通常時は以下だけを表示する。
  - 名前
  - スタック
  - 状態バッジ
  - 見えている `not me`
- 相手の手札裏や細かなラベルは、タップ時の詳細パネルに寄せる。

#### フェーズヘッダー

- 画面上部または中央に、現在フェーズを1行で表示する。
  - 例: `第2ハンド / ベット1: 場札1枚を見て賭けよう`
  - 例: `交換: 自分の not me を変えるか、相手から奪うか選ぼう`
- フェーズが変わった瞬間に短いバナーと音で知らせる。

#### 自分エリア

- 自分の手札2枚と `not me` は最重要情報として維持する。
- ヒントはカード横のラベルではなく、`not me` カードに紐づく「封印タグ」のように見せる。
- `not me` を長押し・タップすると、ヒント履歴や相手リアクションを確認できる。

#### チップ表現

- ベット時はボタンを押した瞬間にチップがポットへ飛ぶ。
- 相手のベットも相手席からポットへチップが飛ぶ。
- ポットは中央に積み上がる表現にし、増えた分だけ一瞬強調する。
- 大勝負は画面揺れ、低音、金色チップなどで心理的重みを出す。

### 完了条件

- 初見プレイヤーが「今何をすればいいか」を3秒以内に理解できる。
- チップの増減とポット移動がラベルではなく動きとして伝わる。

## Phase 3: 交換・奪取インタラクションの刷新

### 目的

本作の差別化要素である `not me` の交換・奪取を、理解しやすく、配信映えする山場にする。

### 実装

#### 交換フェーズのコマンド化

- 単純なモーダルではなく、盤面下部のコマンドドックに3択として表示する。
  - 山札交換: 自分の `not me` を入れ替える。新しいヒントを得る。
  - 奪う: 相手の `not me` を奪う。ただし自分の手札1枚にペナルティ。
  - パス: 現状維持。
- 各選択肢に「得るもの」「失うもの」を短く添える。

#### 奪う対象選択

- `奪う` を押したら、相手席が選択可能状態になる。
- 選べる相手の `not me` カードだけが発光し、それ以外の UI は暗くする。
- 対象選択後、確認画面ではなく「奪取予告」演出にする。

#### 結果演出

- カードが相手席から自分の `not me` 枠へ飛ぶ。
- ペナルティで入れ替わる自分の手札は、before / after を明示する。
- 奪われた側は新しい `not me` とヒントを大きく表示する。
- 奪い返し特殊ケースは専用演出で「おたがい様」を明確にする。

### 完了条件

- 交換・奪取後に、どのカードが誰から誰へ動いたかが一目で分かる。
- プレイヤーがペナルティを理解した上で奪取できる。

## Phase 4: ショーダウンと終了体験の強化

### 目的

「自分の見えなかったカードが開く瞬間」を最大の報酬にし、継続プレイ・共有につなげる。

### 実装

- ショーダウンを段階制にする。
  1. 相手の手札開示
  2. 自分の `not me` が震える
  3. 自分の `not me` 開示
  4. 役に使った4枚が光る
  5. 勝者へチップが流れる
- 役名スタンプと勝敗表示を大きく出す。
- チップ増減を `+120` / `-90` のように明確にする。
- リザルトで、最高の山場を共有文に入れる。
  - 例: `見えない1枚がQで大逆転！ not me で勝利`
- 終了後のホーム復帰と再戦導線を必須にする。

### 完了条件

- 勝敗だけでなく「なぜ勝ったか／負けたか」が視覚的に分かる。
- ゲーム終了後に行き止まりがない。

## Phase 5: ランダムオンライン対戦 MVP

### 目的

ルームコードなしでオンライン対戦を開始できる状態にする。

### サーバー実装

`supabase/functions/online-room/index.ts` に `matchmake` アクションを追加する。

処理方針:

1. `mode = 'queue'` かつ `status = 'waiting'` の部屋を古い順に探す。
2. 自分がすでに参加している待機中 queue ルームがあれば、それを返す。
3. 参加可能な queue ルームがあれば `online_room_players` に空席で参加する。
4. 見つからなければ `mode = 'queue'` の新規ルームを作成し、seat 0 で参加する。
5. 参加後、人数が開始条件を満たしていれば自動開始する。
6. 返却値は既存の `roomId`, `code`, `seat` に加えて `matched`, `players`, `status` を含める。

開始条件 MVP:

- 2人以上で開始可能。
- 将来的には 4人満席、または 15〜30秒待機で2人以上なら開始。
- MVP ではクライアント側の「開始を待つ」ポーリングで十分だが、将来は Realtime に置き換える。

### クライアント実装

- `src/platform/online.ts` に `matchmakeOnlineRoom(displayName)` を追加する。
- `OnlineLobby` に `ランダム対戦` ステップを追加する。
- 待機画面に以下を表示する。
  - 探しています…
  - 現在の参加人数
  - キャンセル
  - 友だちを招待する場合のコード表示
- 一定時間マッチしない場合の選択肢を出す。
  - AI 戦で遊ぶ
  - ルームコードを共有する
  - もう少し待つ

### DB / 運用

- 既存 `online_rooms.mode` を利用するため、MVP では新規テーブル不要。
- ただし stale queue 対策として、後続で以下が必要。
  - `waiting` のまま一定時間更新されない queue ルームを無効化する cleanup。
  - 同一ユーザーの複数 queue 多重参加防止。
  - Edge Function 内の同時 join 競合対策。

### 完了条件

- ホームからランダム対戦を押すだけで、コード入力なしにオンライン部屋へ入れる。
- 2人目以降が来たら自動で対戦が始まる。
- マッチング待機をキャンセルしてホームに戻れる。

## Phase 6: ランダム対戦の商業品質化

### 目的

初期ユーザー数が少ない状態でもランダム対戦を「過疎に見せない」。

### 実装

- 待機時間が長い場合、AI 補充を選択可能にする。
- 内部レーティングまたは初心者保護を導入する。
- 切断時の扱いを定義する。
  - 一定時間無応答なら自動フォールドまたは AI 代打。
  - 途中退出のペナルティは、初期は軽く、ランク実装後に強める。
- 再戦導線を追加する。
  - 同じ相手ともう一度
  - ランダムでもう一度
  - ホームへ戻る
- チップ経済とランキングへの反映方針を決める。
  - 最初は対人オンラインをランキング非反映のカジュアルにする。
  - 不正対策・切断対策が固まってからランク／実チップ反映を解禁する。

### 完了条件

- 過疎時間でもプレイヤーが「失敗した」と感じにくい。
- 再戦・継続プレイへの導線がある。

## 成功指標

- 初回プレイ開始率: ホーム到達者の 70% 以上。
- 初回ゲーム完了率: 50% 以上。
- ランダム対戦開始率: オンライン CTA 押下者の 60% 以上。
- マッチング待機離脱率: 30秒以内で 30% 未満。
- 1セッションあたり平均ハンド数: 8ハンド以上。
- リザルト後の再プレイ率: 35% 以上。
- 共有率: 5% 以上。

## 実装上の注意

- `not me` の秘匿はゲームの核なので、ランダム対戦でも必ず Edge Function の `redactForSeat` 経由で視点別状態だけを返す。
- クライアントにデッキ全体や自分の `not me` の真値を送らない。
- ランダム対戦の queue は同時参加競合が起こりやすいため、満席判定と insert 失敗時のリトライを入れる。
- 初期の対人ランダム対戦は実チップランキングに反映しない。切断・談合・複垢対策が整ってから解禁する。
- UI 刷新は一度に全置換せず、ホーム・リザルト・ゲーム中の順に分割してリリースする。
